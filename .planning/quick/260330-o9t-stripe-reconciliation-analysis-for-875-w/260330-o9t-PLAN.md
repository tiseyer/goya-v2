<task type="auto">
  <name>Task 1: Create and run Stripe reconciliation script</name>
  <files>scripts/stripe-recon-875.ts</files>
  <action>
Create a TypeScript script `scripts/stripe-recon-875.ts` that performs the full reconciliation in sequence. Use `tsx` to run it. The script must:

**Step 1 — Fetch WooCommerce subscriptions without Stripe IDs:**
- Paginate through WooCommerce subscriptions API: GET /wp-json/wc/v3/subscriptions?status=active&per_page=100&page=N
- Auth: Basic auth with WOO_CONSUMER_KEY:WOO_CONSUMER_SECRET
- For each subscription, check meta_data for `_stripe_subscription_id` — keep only those where it is empty/missing/null
- Also filter out customers whose meta contains wp_roles with "faux" or "robit" (check customer meta or fetch customer details if needed)
- Collect: subscription_id, customer_id (= wp_user_id), customer email, line_items[0] product_id and name, billing_period, start_date, next_payment_date, and payment meta (_stripe_customer_id, _stripe_subscription_id, _payment_method from meta_data)
- Save to /tmp/wc_no_stripe_subs.json
- Log count found

**Step 2 — Look up each customer in Stripe:**
- For each unique email from Step 1, call Stripe API: GET /v1/customers?email={email}&limit=10
- Use STRIPE_SECRET_KEY as Bearer token
- Add 100ms delay between calls to avoid rate limits
- For each Stripe customer found, also fetch:
  - Subscriptions: GET /v1/subscriptions?customer={cus_id}&limit=10 (get id, status, plan, current_period_end, cancel_at_period_end)
  - Recent charges: GET /v1/charges?customer={cus_id}&limit=3 (status, amount, created)
  - Check if default_source or invoice_settings.default_payment_method exists
- Handle multiple Stripe customers per email (flag as anomaly)

**Step 3 — Categorize into buckets:**
- A) "active_in_stripe": Stripe customer exists AND has at least one subscription with status=active
- B) "cancelled_in_stripe": Stripe customer exists, has subscription(s) but all are cancelled/canceled
- C) "no_stripe_customer": No Stripe customer found for this email
- D) "manual_renewal": Stripe customer exists, subscription exists but latest invoice amount_paid=0 or subscription plan amount=0
- E) "failed_payment": Stripe customer exists, subscription in incomplete/unpaid/past_due state

Flag anomalies:
- Multiple Stripe customers for same email
- Stripe subscription product differs from WooCommerce product
- Cancellation within last 30 days

**Step 4 — Cross-reference with Supabase:**
- Use Supabase REST API (https://snddprncgilpctgvjukr.supabase.co/rest/v1/profiles) with service role key from SUPABASE_SERVICE_ROLE_KEY env var
- For each email, check if profile exists and if stripe_customer_id is populated
- If stripe_customer_id exists in Supabase but WooCommerce has no stripe sub ID, flag as "stripe_id_in_supabase_only"

**Step 5 — Generate outputs:**
- Save full categorized data to /tmp/stripe_recon_875.json (array of objects with wc_data, stripe_data, bucket, anomalies, supabase_match)
- Generate /tmp/recon_summary.md with:
  - Total count
  - Count per bucket (A/B/C/D/E) with percentages
  - Migration strategy per bucket
  - Anomaly summary
  - List of "stripe_id_in_supabase_only" cases

Use fetch() for all HTTP calls (no external deps needed with tsx). Read env vars from process.env (they should be in .env.local or shell env).

If .env.local exists, load it manually by reading the file and parsing KEY=VALUE lines into process.env before starting.
