flows table columns:
  id uuid PK, name text NOT NULL, description text, status text (draft|active|paused|archived),
  priority integer DEFAULT 0, display_type text (modal|fullscreen|top_banner|bottom_banner|notification),
  modal_dismissible boolean, modal_backdrop text (blur|dark|none),
  trigger_type text (login|manual|page_load), trigger_delay_seconds integer,
  frequency text (once|every_login|every_session|custom),
  conditions jsonb DEFAULT '[]', schema_version integer DEFAULT 1,
  is_template boolean DEFAULT false, template_name text,
  created_by uuid, created_at timestamptz, updated_at timestamptz

flow_steps table columns:
  id uuid PK, flow_id uuid FK->flows ON DELETE CASCADE,
  position integer DEFAULT 0, title text,
  elements jsonb DEFAULT '[]', schema_version integer DEFAULT 1, created_at timestamptz

flow_branches table columns:
  id uuid PK, step_id uuid FK->flow_steps ON DELETE CASCADE,
  element_key text NOT NULL, answer_value text NOT NULL,
  target_step_id uuid FK->flow_steps ON DELETE CASCADE,
  UNIQUE(step_id, element_key, answer_value)

flow_responses table columns:
  id uuid PK, flow_id uuid FK, user_id uuid FK,
  status text (in_progress|completed|skipped|dismissed),
  started_at timestamptz, completed_at timestamptz,
  last_step_id uuid FK, responses jsonb DEFAULT '{}',
  UNIQUE(flow_id, user_id)

flow_analytics table columns:
  id uuid PK, flow_id uuid FK, user_id uuid FK,
  event text (shown|started|step_completed|completed|skipped|dismissed),
  step_id uuid FK, created_at timestamptz

    **lib/flows/types.ts** — Create TypeScript interfaces mirroring the 5 flow builder tables:

    1. `Flow` interface with all 17 columns from the flows table. Use string literal unions for status (`'draft' | 'active' | 'paused' | 'archived'`), display_type, trigger_type, frequency. Conditions is `FlowCondition[]`. Timestamps are `string` (ISO format from Supabase).

    2. `FlowStep` interface with id, flow_id, position, title, elements (typed as `FlowElement[]`), schema_version, created_at.

    3. `FlowElement` interface — discriminated union on `type` field. Types: `'info_text' | 'short_text' | 'long_text' | 'single_choice' | 'multi_choice' | 'dropdown' | 'image_upload' | 'image' | 'video'`. Common fields: `type`, `element_key` (string), `label` (string), `required` (boolean), `help_text` (string | null). Type-specific: choices elements get `options: { label: string; value: string }[]`, image gets `src: string; alt: string`, video gets `url: string`, info_text gets `content: string`.

    4. `FlowBranch` interface with id, step_id, element_key, answer_value, target_step_id.

    5. `FlowResponse` interface with all columns from flow_responses.

    6. `FlowAnalyticsEvent` interface with all columns from flow_analytics.

    7. `FlowCondition` interface: `{ type: FlowConditionType; operator: string; value: string | string[] | boolean | number }` where `FlowConditionType = 'role' | 'onboarding_status' | 'has_profile_picture' | 'subscription_status' | 'birthday' | 'flow_completed'`.

    8. Input types for mutations:
       - `CreateFlowInput`: Pick name, description, status, priority, display_type, modal_dismissible, modal_backdrop, trigger_type, trigger_delay_seconds, frequency, conditions, is_template, template_name — all optional except name
       - `UpdateFlowInput`: Partial of CreateFlowInput (all optional)
       - `CreateStepInput`: Pick flow_id, position, title, elements — flow_id required
       - `UpdateStepInput`: Partial of CreateStepInput minus flow_id
       - `UpsertBranchInput`: { step_id, element_key, answer_value, target_step_id }

    9. `FlowWithSteps` type: `Flow & { steps: (FlowStep & { branches: FlowBranch[] })[] }` — used by getFlowWithSteps

    Export all types. Add `'use server'` is NOT needed — these are plain types, no directive.

    **lib/flows/flow-service.ts** — Admin CRUD for flows table:

    1. `import 'server-only'` at top.
    2. Import `getSupabaseService` from `@/lib/supabase/service`.
    3. Import types from `./types`.
    4. Follow existing pattern: `const supabase = getSupabaseService() as any;`

    Functions (all return `{ data, error }` pattern like users.ts):

    - `listFlows(params: { status?: Flow['status']; is_template?: boolean })` — select all from flows, filter by status and is_template if provided, order by priority DESC then created_at DESC. No pagination needed for admin (flows count is low).

    - `getFlowById(id: string)` — select single flow by id.

    - `getFlowWithSteps(id: string)` — select flow by id, then select all flow_steps for that flow_id ordered by position ASC, then select all flow_branches where step_id IN the step IDs. Assemble into `FlowWithSteps` shape by attaching branches to their parent steps. Return `{ data: FlowWithSteps | null, error }`.

    - `createFlow(input: CreateFlowInput, createdBy: string)` — insert into flows with input fields + created_by. Return the inserted row with `.select().single()`.

    - `updateFlow(id: string, input: UpdateFlowInput)` — update flows where id matches. Use `.select().single()`. Do NOT include updated_at — the DB trigger handles it.

    - `deleteFlow(id: string)` — delete from flows where id matches. CASCADE handles steps, branches. Return `{ error }`.

    - `duplicateFlow(id: string, createdBy: string)` — call getFlowWithSteps, then createFlow with name `"${original.name} (Copy)"` and status `'draft'`, then create all steps and branches for the new flow. Return the new flow.

    **lib/flows/step-service.ts** — Admin CRUD for flow_steps and flow_branches:

    1. Same imports pattern as flow-service.
    2. Functions:

    - `listSteps(flowId: string)` — select from flow_steps where flow_id, order by position ASC. Return `{ data, error }`.

    - `getStepById(id: string)` — select single step with its branches (separate query on flow_branches where step_id). Return step with branches attached.

    - `createStep(input: CreateStepInput)` — insert into flow_steps. Return inserted row.

    - `updateStep(id: string, input: UpdateStepInput)` — update flow_steps where id. Return updated row.

    - `deleteStep(id: string)` — delete from flow_steps where id (CASCADE deletes branches). Return `{ error }`.

    - `reorderSteps(flowId: string, stepIds: string[])` — for each stepId in the array, update position to its index. Use a loop of individual updates (flow step count is small, typically <20). Return `{ error }`.

    - `upsertBranches(stepId: string, branches: UpsertBranchInput[])` — delete all existing branches for stepId, then insert new ones. This is a replace-all operation (simpler than diffing). Return `{ data, error }`.

    **lib/flows/cycle-detection.ts** — DFS cycle detection on branch graph:

    1. Import `FlowBranch` from `./types`.
    2. Export function `detectCycle(steps: { id: string }[], branches: FlowBranch[]): { hasCycle: boolean; cyclePath?: string[] }`.
    3. Build adjacency list: for each branch, add edge from `branch.step_id` to `branch.target_step_id`.
    4. Run DFS from each unvisited node. Track `visiting` (gray) and `visited` (black) sets.
    5. If a node is encountered while in `visiting` state, a cycle exists. Reconstruct the cycle path from the recursion stack.
    6. Return `{ hasCycle: false }` if no cycle found, or `{ hasCycle: true, cyclePath: ['step-A', 'step-B', 'step-C', 'step-A'] }` if cycle found.
    7. This is pure logic — no Supabase dependency, no `server-only` needed. ~30 lines.
