Output: One new actions module, updates to engine and respond route.

**Main export:**

```typescript
export async function executeStepActions(params: {
  flowId: string;
  userId: string;
  stepId: string;
  actions: FlowStepAction[];
  answers: Record<string, unknown>;
  userEmail: string;
}): Promise<ActionResult[]>
```

Define locally:
```typescript
interface ActionResult {
  type: FlowActionType;
  success: boolean;
  skipped?: boolean;  // true if idempotency check prevented execution
  error?: string;
  data?: Record<string, unknown>;  // e.g., stripe checkout URL
}
```

**Idempotency check (run BEFORE each action):**
1. Query `flow_action_executions` for `(flow_id, user_id, step_id, action_type)`.
2. If row exists, return `{ type, success: true, skipped: true }` — do NOT execute.
3. After successful execution, INSERT into `flow_action_executions`.
4. Use a try/catch on the insert — if UNIQUE violation (race condition), that is fine, the action already ran.

**Per-action handlers (implement as private async functions):**

**`save_to_profile`** (per D — ACTION-02):
- `config: { mappings: Record<string, string> }` where key = element_key, value = profile column name
- For each mapping, read `answers[element_key]` and build an update object
- Upsert: `supabase.from('profiles').update(updateObj).eq('id', userId)`
- Return success/error

**`send_email`** (ACTION-01 — stub for v1):
- `config: { template_id?: string, subject?: string }`
- Log a console.warn: `"send_email action not implemented — email template: ${config.template_id}"`
- Return `{ success: true }` — stub, no actual email sending in v1
- NOTE: This is intentionally a no-op stub. Real email integration is deferred.

**`kit_tag`** (per D — ACTION-03):
- `config: { tag_id: string }`
- Call `tagSubscriber(userEmail, config.tag_id)` from `@/lib/flows/kitcom`
- Return the result directly

**`stripe_checkout`** (per D — ACTION-04):
- `config: { price_id: string, success_url?: string, cancel_url?: string }`
- Deterministic idempotency key: `flow_${flowId}_${stepId}_${userId}` (per CONTEXT.md decision)
- Call `getStripe().checkout.sessions.create({ mode: 'payment', line_items: [{ price: config.price_id, quantity: 1 }], customer_email: userEmail, success_url: config.success_url || `${process.env.NEXT_PUBLIC_APP_URL}/dashboard`, cancel_url: config.cancel_url || `${process.env.NEXT_PUBLIC_APP_URL}/dashboard`, idempotency_key })` — pass idempotency key via the Stripe request options (second arg): `{ idempotencyKey }`
- Return `{ success: true, data: { checkout_url: session.url } }` or error

**`redirect`** (ACTION-01):
- `config: { url: string }`
- No server-side execution needed — return `{ success: true, data: { redirect_url: config.url } }` for the client to handle

**`trigger_flow`** (per D — ACTION-05):
- `config: { target_flow_id: string }`
- Loop detection: Accept optional `depth` param (default 0). If `depth >= 3`, return error "Maximum flow trigger depth exceeded".
- Check if user already has a flow_response for target_flow_id. If so, skip (return skipped: true).
- Insert new flow_response: `{ flow_id: target_flow_id, user_id: userId, status: 'in_progress', started_at: new Date().toISOString(), responses: {} }`
- Return success

**`success_popup`** (ACTION-01):
- `config: { title?: string, message?: string }`
- No server-side execution — return `{ success: true, data: { popup: { title: config.title, message: config.message } } }` for client rendering

**`mark_complete`** (ACTION-01):
- Import and call `completeFlow(userId, flowId)` from `./engine`
- Return success/error

**Error handling strategy (per CONTEXT.md — Claude's discretion):**
Execute all actions sequentially (per CONTEXT decision: synchronous await all). If one action fails, continue executing the remaining actions. Collect all results. The caller receives the full ActionResult[] array and can inspect which succeeded/failed. Do NOT roll back successful actions on partial failure.
