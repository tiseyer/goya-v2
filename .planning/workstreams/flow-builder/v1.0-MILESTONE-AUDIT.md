# Milestone v1.0 Flow Builder — Audit Report

## Summary

**Status: GAPS FOUND**

All 7 phases completed and individually verified. However, cross-phase integration check reveals that the **actions engine and analytics recording are completely unwired at runtime** — the components exist in isolation but the data flow between them is broken.

## Critical Issues

### 1. Actions Engine Completely Dark (ACTION-01 through ACTION-05)

The actions engine (`lib/flows/actions.ts`) with all 8 handlers exists and has idempotency protection, but it is **never called at runtime**. The break chain:

1. `FlowStep` TypeScript type has no `actions` field
2. `getFlowWithSteps` doesn't expose step actions to the client
3. `FlowPlayer.handleNext` sends `actions: undefined` in POST body
4. `StepActionsEditor` saves to Zustand only — step PATCH never includes `actions`
5. `executeStepActions` is never reached

**Fix requires:** 4 coordinated changes across types, services, API, and player.

### 2. Analytics Source Data Absent (ANALYTICS-01, ANALYTICS-02)

The analytics dashboard is fully built (API route, metric cards, Recharts chart, time filters) but the **player never records analytics events**. Only `completed` is written by `completeFlow()`. Events `shown`, `started`, `step_completed`, `dismissed`, `skipped` are never inserted.

**Fix requires:** Analytics recording calls in FlowPlayer + server-side recording in engine.

### 3. Profile Mappings & Step Actions Not Persisted (ADMIN-10, ADMIN-12)

Both profile field mapping and step actions are configured in the admin UI but stored only in Zustand client state. They are lost on page reload and never reach the database.

**Fix requires:** Include `actions` and `mappings` in step PATCH payload, add fields to FlowStep type.
